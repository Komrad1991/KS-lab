# Условия работы над задачей: Command Palette для TiniJS

Цель проекта: реализовать полнофункциональную Command Palette для фреймворка TiniJS с поддержкой горячих клавиш, контекстной фильтрации и интеграции с роутером.

1. Стек и базовые конвенции
- Язык: TypeScript.
- Базовые классы: TiniComponent из @tinijs/core.
- UI-компоненты: @tinijs/ui (Input, Dialog, Spinner).
- Хранилище состояния: createStore из @tinijs/store.
- Экспорт: public-api.ts должны раскрывать новые модули/сервисы.
- Поддерживаемые темы: светлая/темная; стили под 9 тем оформления.

2. Правила разработки и тестирования
- Не изменять существующий код до утверждения архитектуры.
- Пишем unit-тесты для новых модулей и интеграционные тесты по мере необходимости.
- Тестируем fuzzy search отдельно как модуль.
- Примеры использования размещаем в apps/ (один приложение пример).

3. Архитектурные принципы
- Разделение: UI-поведение палитры отдельно от регистрации команд (CommandService).
- Контекстные команды зависят от текущего состояния приложения и маршрута.
- Асинхронность: команды могут быть асинхронными; поддерживаются индикаторы загрузки.

4. Интеграция с роутером
- Автоматическая регистрация навигационных команд для зарегистрированных роутов.
- Команды навигации формируются как "Перейти к [Название]"; поддержка пути через @tinijs/router.

5. Поддержка тем и стилизации
- Палитра и элементы поддерживают светлую/темную тему; стиль интегрируется с 9 стилевыми темами.
- Использовать существующие средства стилизации UI-пакета.

6. Безопасность и устойчивость
- История команд хранится в localStorage; обрабатываем ошибки доступа.
- Поддерживаем отмену выполнения команд.
- Предусматриваем откат действий, если команда не подходит по контексту.

7. Документация и поддержка
- API-справка и примеры использования, отдельные разделы документов.
- Комментарии в архитектурных файлах и патчах.

8. Разграничение ролей и гранулярности
- Архитектура разделена на слои UI и бизнес-логики; промис-ориентированный API.
- Прототипирование и детальная спецификация согласовываются на этапе планирования.

Примечание: детали реализации утверждаются на этапе прототипирования; этот документ задаёт направление и контекст.
