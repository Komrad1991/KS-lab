# План реализации: Command Palette для TiniJS

Этот план охватывает все пункты ТЗ по Command Palette, включая базовую палитру, регистрацию команд, систему hotkeys, интеграцию с роутером, UI-компоненты, дополнительные функции, тестирование и документацию.

Цель проекта
- Реализовать полнофункциональную Command Palette с быстрым поиском, навигацией, исполнением команд, историей использования и поддержкой горячих клавиш. Палитра должна работать с роутером и поддерживать 9 тем оформления.

1. Этапы реализации (между этапами есть зависимости)

Этап 0. Подготовка и выравнивание требований (1-3 дня)
- Уточнить и зафиксировать архитектуру и контракты (API CommandService, типы CommandDefinition, ShortcutDefinition).
- Обновить документацию API и спецификацию интерфейсов.
- Согласовать формат и форматирование артефактов (docs/backlog/test-plan).

Этап 1. Архитектура и контракт API (2-4 дня)
- Спроектировать CommandService в пакете @tinijs/toolbox: registerCommand, unregisterCommand, getRegisteredCommands, groupings, shouldShow, subscribeContext.
- Определить модели данных: CommandDefinition, ShortcutDefinition, Category.
- Определить контракт по контексту и маршрутизации: как работать с контекстом страницы и списком роутов.
- Определить интерфейс взаимодействия между UI и бизнес-логикой.
- Протокольное решение по экспорту через public-api.ts.

Этап 2. UI-компоненты (2-6 дней)
- Реализовать компоненты: tini-command-palette, tini-command-item, tini-shortcut в @tinijs/ui.
- Реализовать оверлей/модальное окно, поле поиска с fuzzy search, список команд, индикатор загрузки.
- Реализовать визуальное отображение соотношений (иконки, названия, описания, shortcut).
- Обеспечить поддержку 9 тем оформления и светлую/темную тему.
- Встроить локальное хранение истории использования через localStorage.

Этап 3. Регистрация команд и контексты (3-5 дней)
- Реализовать API регистрации команд: registerCommand(id, name, description, icon, action, keywords?).
- Поддержка группировки по категориям; динамическая регистрация/удаление.
- Поддержка условий отображения команд (shouldShow(context)).
- Поддержка контекст-ориентированных команд, подписки на контекст.
- Интеграция с маршрутизатором: автоматическая регистрация навигационных команд на основе зарегистрированных роутов; формат "Перейти к [Название]".

Этап 4. Горячие клавиши (2-4 дня)
- Реализация registerShortcut(keys, commandId, preventDefault?) с поддержкой модификаторов и последовательностей.
- Обработка конфликтов горячих клавиш; выбор приоритетной команды.
- Отображение назначенных сочетаний в палитре рядом с командами через tini-shortcut.

Этап 5. Поиск и релевантность (2-4 дня)
- Реализация fuzzy search (возможность transliterate как опция).
- Релевантность: соответствие текста имени/описания/keywords, плюс история использования для ранжирования.
- Подсветка совпадений и автодополнение аргументов команд (если применимо).

Этап 6. История, аргументы и асинхронность (2-4 дня)
- История команд: сохранение и чтение из localStorage (tinijs.history.commands).
- Поддержка аргументов команд и асинхронных действий: индикатор загрузки, отмена выполнения.
- Обработка ошибок и rollback (если команда не удаётся выполнить из-за контекста).

Этап 7. Тестирование (2-6 дней, параллельно по задачам)
- Unit-тесты для CommandService, регистрации/удаления, shouldShow, контекст и навигация.
- Тесты for fuzzy search алгоритма и transliterate.
- Тесты клавиатурных событий (ArrowUp/Down, Enter, Esc, последовательности).
- Интеграционные тесты: регистрация роутов и навигация через палитру.
- Пример использования в apps/ и тестовые сценарии.

Этап 8. Документация и примеры (2-3 дня)
- Обновление API-доков: API_COMMANDS.md, TINI_COMMAND_PALETTE_SPEC.md и UI_COMPONENTS.md.
- Пример использования в apps/; документация по API и примеры использования.
- Руководство по сборке и тестированию (AGENTS.md обновление).

Этап 9. Финальная сборка и релиз (1-2 дня)
- Запуск полного цикла: lint, typecheck, тесты, сборка.
- Подготовка PR с описанием изменений и ключевых решений.

2. Критерии приемки (Acceptance Criteria)
- Палитра открывается Cmd/Ctrl+K, поиск работает, стрелки навигации, Enter выполняет.
- Поддерживаются зарегистрированные hotkeys; конфликт-решение.
- Роутер-интеграция: навигационные команды автоматически зарегистрированы, формат "Перейти к [Название]".
- История команд хранится и используется для подсказок.
- Поддержка светлой/темной темы и 9 тем оформления.
- Асинхронные команды и отмена выполнения; индикаторы загрузки.
- Пример использования в apps/ присутствует.

3. Метрики успеха
- Покрытие тестами: не менее 80% по критичным путям.
- Уровень ошибок в релизе: ноль критичных ошибок в первой итерации.
- Время отклика палитры на открытие не более 150–250 мс при малом списке.

4. Deliverables
- План проекта и backlog; патчи к docs/PLAN_COMMAND_PALETTE.md и др.
- Реализация кода: UI, сервисы, маршрутизация, тесты.
- Документация API и примеры использования.
- Demo-пример в apps/.

5. Риски и меры
- Неполные требования/изменения в ТЗ: регулярные синхронизации с заказчиком.
- Конфликты горячих клавиш: поддержка приоритетности и уведомления.
- Сложности с fuzzy-search и производительностью: оптимизация и ограничение индекса.

6. Пример графика
- Недели 1-2: дизайн API, планирование; недели 3-4: реализовать Core UI и CommandService; недели 5-6: реализоватьHotkeys, router integration; недели 7-8: тесты и документация; неделя 9: финальная сборка и релиз.

Примечание: этот план — ориентир. Приоритеты и сроки могут быть скорректированы по согласованию с командами.`
